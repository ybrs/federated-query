# Code-Quality Review - 2026-06-28

Scope: full read of `federated_query/`. Headline: the binder and the
`plan/emit/*` + `plan/expressions.py` shared-helper layer are genuinely
consolidated and fail-loud (allowlist+raise). The disease persists mainly in the
optimizer/physical layers: catch-all "return node unchanged" / magic-default
walkers, type inference by guessing, and pushdown/source-compatibility decisions
reasoned in multiple places that disagree.

## A. SILENT FAILURES (highest priority)

- `datasources/postgresql.py:331-332` - `except adbc_driver_manager.Error: pass`
  in `_bound_adbc_batch_size`. Genuine catch-and-swallow ("best-effort if
  unsupported"). Violates rule #1. Fix: let it raise, or gate on a capability
  check, not a swallowed exception.
- `optimizer/decorrelation.py:1140-1148 / 1533-1553` -
  `_rewrite_inner_join_condition` applies every subquery conjunct against the
  LEFT input only; the docstring admits a right-side correlation "whose name
  overlaps a left column binds silently to the left (a known limitation, not yet
  rejected)." Manufactures wrong rows for a legal query - the EPIC-FAIL case.
  Fix: a correlation not provably bound to the chosen side must raise.
- `optimizer/decorrelation.py:245-264` - `_plan_expressions`/
  `_other_plan_expressions` is a denylist walker that `return []` for any node
  not Filter/Projection/Aggregate/Sort/Join/Values. Backs `_is_correlated`, the
  `_validate_no_outer_left` safety net, and the final
  `_raise_if_subquery_expression` guard. `Scan` (carries
  `filters`/`aggregates`/`order_by_keys`) and `GroupedLimit`/`SingleRowGuard`
  are silently ignored, so a correlated ref or surviving subquery inside
  `Scan.filters` defeats all three guards - contradicting the module's "never
  silently skips a subquery" claim. Fix: allowlist+raise on unhandled node type.
- `optimizer/decorrelation.py:1596-1601 + 195-201` - control flow by
  substring-matching an exception message
  (`"equality correlation predicates can cross" in str(error)`), with the
  trigger string duplicated at the raise sites (776, 1024) and paraphrased at
  902. Any `DecorrelationError` from deep inside whose message happens to
  contain the substring is rerouted to the LATERAL fallback for the wrong
  reason. Fix: a typed exception/flag, not message text.
- `parser/parser.py:713-738` - `_extract_join_type` returns `JoinType.INNER` for
  any unrecognized `side`/`kind`. A SEMI/ANTI/unknown side is silently
  downgraded to INNER = wrong join. Fix: raise on unrecognized side/kind.
- `parser/parser.py:1939-1947` - `_append_function_arg` silently drops any
  function-arg slot that is not an `exp.Expression` or `list`. `_convert_trim`
  (1368) hand-guards the one known case and admits "the generic collector
  silently skips" it - the banned denylist+guard pattern. Fix: raise on an
  unexpected slot.
- `parser/parser.py:2004-2032` - `_rewrite_expression` (HAVING
  aggregate->output remap) handles only aggregate FunctionCall / BinaryOp /
  UnaryOp; final `return expr` does not recurse into
  CaseExpr/Between/InList/Cast/Extract/Window/Tuple. `HAVING CASE WHEN
  SUM(x)>0 ...` is never remapped. Incomplete walker; cover all or raise.
- `parser/parser.py:869-881` - `_has_aggregate_functions` inspects only
  `select.expressions`, missing aggregates that appear only in HAVING/ORDER BY
  (`... HAVING COUNT(*)>1`). No Aggregate node is built -> mis-plan over raw
  scan.
- `optimizer/rules.py:825-832` - `_propagate_sort_metadata` calls
  `self._push_to_scan(...)` and discards its return (returns a new scan, mutates
  nothing) - a silent no-op; and it is only ever called by itself (dead). Line
  830 also duck-types via `getattr(node,"input",None)`.
- `optimizer/cost.py:46-68, 289, 363` - `estimate_cardinality`/cost estimators
  return magic `1000`/`1000.0` for unhandled node types (Sort/Union/
  SetOperation). A Sort over a 1M-row scan is costed at 1000 rows. Default
  should raise, not fabricate.
- `executor/executor.py:54-65` - `_get_merge_engine` uses `getattr(self.config,
  ...)` defaults because `self.config` "is an ExecutorConfig in the real runtime
  but a Catalog in the decorrelation test harness." Silent wrong-type acceptance
  and test-accommodation in production code. Fix the abstraction; raise on wrong
  config type.
- `executor/expression_evaluator.py:126-132` - `_eval_column`: when `expr.table`
  is set but `(table,column)` is missing from `alias_map`, it silently falls
  back to bare `self.batch.column(expr.column)` (wrong namespace) instead of
  raising.
- `catalog/catalog.py:153-205` - `_map_type`/`_map_textual_type` default any
  unknown DB type string to `VARCHAR` (TODO at line 52). Silent mistype. The
  ClickHouse and Postgres connectors correctly RAISE on unknown types - this is
  the inconsistent lenient one.
- `processor/query_executor.py:132-137` - `_run_physical_plan`: `if plan.format
  == ExplainFormat.JSON:` with no else; any other `ExplainFormat` silently falls
  through. Guard or raise.

## B. SEMANTIC DUPLICATION (clusters doing one job)

1. Type inference, 2 divergent implementations: `physical.py:805-814`
   `PhysicalProjection._infer_expression_type` (correct, evaluates empty batch)
   vs `physical.py:1665-1699` `PhysicalHashAggregate._infer_output_type`/
   `_infer_aggregate_type` (guesses, and is wrong - see C). Collapse onto the
   evaluate-empty-batch approach.
2. "Available/required column names for a plan node", 2 disagreeing copies:
   `rules.py:336 _get_column_names` (handles Join, Aggregate) vs `rules.py:969
   _get_available_columns` (does not handle Join) - root of the ORDER-BY-
   pushdown bug. Should be one helper in `optimizer/pushdown.py`.
3. Join output schema / aliases, identical duplicate: `physical.py:1221-1247`
   (PhysicalHashJoin.schema/column_aliases) vs `physical.py:1515-1541`
   (PhysicalNestedLoopJoin) are byte-identical. Plus join-keyword map rebuilt at
   `physical.py:1162-1172` despite module-level `_MERGE_JOIN_KEYWORDS`.
4. Merge-join SQL assembly: `physical.py:1152-1160 _join_sql` vs
   `physical.py:1499-1513 _merge_sql` build the same string.
5. Select-list-render-and-join: `physical.py:1360-1368`, `1588-1598`, `898-916`
   - three near-identical builders.
6. Source-compatibility decision, 3+ places that DISAGREE:
   `single_source_pushdown.py:800 _source_compatible` (name-only) vs
   `physical_planner.py:807 _is_remote_join_candidate` (name + connection
   identity) vs inlined copies at `single_source_pushdown.py:511, 726, 749`
   (differ on the `datasource is None` CTE branch). Name-only matching can merge
   two sources sharing a name but not a connection -> wrong results.
7. "Collect all scans" tree-walk duplicated: `physical_planner.py:210
   _lateral_base_scans` vs `521 _collect_base_scans` (identical), overlapping
   `_leftmost_scan` (226) and `_collect_relation_aliases` (748).
8. Ordered-key + NULLS-default logic in 2 files: `plan/emit/clauses.py:45
   _ordered_key` vs `plan/emit/expressions.py:253 SqlglotEmitter._ordered_key`.
9. Set-op keyword build duplicated: `single_source_pushdown.py:491` and `:557`.
10. Clause assembly half-delegated: `single_source_pushdown.py:909-974`
    hand-builds SELECT/WHERE/LIMIT/OFFSET as f-strings while `emit/clauses.py`
    already provides `select_expressions`/`apply_limit_offset`. Route through it.
11. Correlation analysis duplicated (one dead): `decorrelation.py:89-176
    CorrelationAnalyzer` (+ `_collect_tables` 115, `_is_outer_ref` 165) is a
    second full correlation walker used only by tests, never by `Decorrelator`;
    duplicates `_collect_inner_aliases` (283) and the live `_references_outer`
    (720). Equi-correlation guard implemented 3x (772, 897, 1013).

## C. CORRECTNESS / SHIPPING BUGS

- `physical.py:1694-1697` - `_infer_aggregate_type` maps SUM/AVG/MIN/MAX all to
  `float64`. MIN/MAX must preserve arg type; SUM of ints is not float64. Because
  `_execute_merge_aggregate` (1576) force-casts DuckDB's real output to this
  declared schema, `MIN(varchar)` declared float64 corrupts data or errors on
  cast. Real wrong-result/cast-failure.
- `physical_planner.py:644-649` - unreachable block after `return` at line 642
  inside `_is_column_literal_eq`, referencing `left_plan`/`right_plan`/`join`
  not in scope; copy-paste cruft that would `NameError` if reached. Delete.
- `physical_planner.py:732-740` - `_orient_by_name` falls back to keeping the
  original key order when it cannot determine which join side a key belongs to.
  A hash join then keys on mismatched columns -> wrong/empty results. Should
  raise when orientation is unresolvable.
- `physical.py:380` - `_qualify_join_condition` leaves any expr that is not
  ColumnRef/BinaryOp/UnaryOp unchanged, so a FunctionCall/Cast/InList in an ON
  clause doesn't get its columns `l.`/`r.`-qualified -> ambiguous/wrong ON.
- `physical.py:795 / 1671-1673` - `PhysicalProjection.schema` and
  `_infer_output_type` resolve a qualified ColumnRef with
  `input_schema.get_field_index(expr.column)` using the BARE name, ignoring
  `expr.table`/alias_map. With two same-named columns `get_field_index` returns
  -1 and `.field(-1)` silently picks the last field -> declared schema disagrees
  with executed output.
- `rules.py:1002` - `OrderByPushdownRule._get_available_columns` returns empty
  set for a `Join` child; for 3+-way joins `columns_belong_to_side` mis-decides
  and ORDER BY can push to the wrong side. (Caused by duplication #2.)
- `rules.py:469, 511` - projection-pushdown silently skips `Sort`:
  `_collect_required_columns` does not collect sort-key columns nor recurse into
  `Sort.input`; `_prune_columns` returns it unchanged. Latent column-pruning
  hazard.
- `cost.py:333-338` - `_estimate_limit_cost` computes `input_cost` then never
  uses it; a LIMIT is costed as if its child were free.
- Union vs SetOperation handled by different subsets of rules: predicate/
  aggregate recurse `SetOperation` only (rules.py:115, 1230); limit recurses
  `Union` only (581); order-by handles both (808, 1079); single_source handles
  `SetOperation` only. Two parallel set-op representations; optimization
  silently depends on which node the binder produced. Likely one is legacy.
